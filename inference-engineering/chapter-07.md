## 容器化：打包你的推理服务

第一份文档：《容器化入门》。

**为什么要容器化**

容器把应用和依赖打包在一起。"在我机器上能跑"的问题不再存在。

**Docker术语**

| 术语 | 含义 |
|-----|-----|
| 容器（Container） | 运行中的隔离环境 |
| 镜像（Image） | 可执行的软件包 |
| Dockerfile | 创建镜像的指令文件 |
| 注册表（Registry） | 存储和分发镜像的仓库 |

**镜像分层**

Docker镜像由层组成：

```
┌─────────────────────┐
│   容器层（临时）     │ ← 运行时创建，容器停止就消失
├─────────────────────┤
│   你的代码/配置     │
├─────────────────────┤
│   推理引擎/依赖     │
├─────────────────────┤
│   基础镜像(Ubuntu)  │
└─────────────────────┘
```

vLLM、SGLang都有官方基础镜像。从这些开始比从头构建更可靠。

**依赖管理**

推理的依赖链**长且脆弱**：CUDA版本、PyTorch版本、引擎版本——任何一个不匹配就可能出问题。

关键实践：
- **锁定版本**：每个依赖指定精确版本
- **精简打包**：只包含必要的依赖，镜像越小启动越快
- **保存好的构建**：一旦能工作，就把镜像保存起来

新模型发布时尤其要注意——第一天的依赖可能是nightly版本，几天后要重新构建稳定版。

## 自动扩缩容

文档二：《让资源跟上流量》。

**目标**

始终有足够的资源服务请求，保持延迟SLA，同时不浪费钱在空闲GPU上。

**Kubernetes**

容器编排系统。管理在哪里运行多少个副本。

```
        ┌─────────────────┐
        │    控制平面     │ ← 做路由和扩缩决策
        └───────┬─────────┘
                │
    ┌───────────┼───────────┐
    │           │           │
┌───┴───┐   ┌───┴───┐   ┌───┴───┐
│ 副本1 │   │ 副本2 │   │ 副本3 │  ← 工作平面
└───────┘   └───────┘   └───────┘
```

**扩缩依据**

两种信号：

| 信号 | 特点 |
|-----|-----|
| 利用率 | 滞后指标，反映当前状态 |
| 流量 | 可前瞻，但不一定反映负载 |

结合使用效果最好。

**五个关键配置**

1. **最小副本数**：无论流量多低，至少保持几个副本
2. **最大副本数**：流量高峰时的上限
3. **扩缩窗口**：用多长时间的数据做决策
4. **缩容延迟**：流量下降后等多久再缩容（防止震荡）
5. **并发目标**：每个副本能处理多少并发请求

## 批处理与并发

**三种批处理方式**

| 方式 | 说明 |
|-----|-----|
| 静态批处理 | 等批次满了再开始 |
| 动态批处理 | 满了或超时就开始 |
| 连续批处理 | token级别调度，一直在跑 |

vLLM、SGLang、TensorRT-LLM都实现了连续批处理。

**批大小权衡**

- 批大小↑ → 吞吐量↑，但延迟↑
- 批大小↓ → 延迟↓，但吞吐量↓

测试找到你的最佳平衡点。

## 冷启动

新副本启动的时间。冷启动慢，就不敢缩容，就会浪费钱。

**冷启动的四个阶段**

```
GPU分配 → 镜像加载 → 模型加载 → 引擎启动
```

每个阶段都可以优化：

| 阶段 | 优化策略 |
|-----|---------|
| GPU分配 | 预热节点池，和云商谈判 |
| 镜像加载 | 精简镜像，本地缓存 |
| 模型加载 | 量化减小体积，高速存储 |
| 引擎启动 | 缓存编译结果 |

**模型加载是关键**

几百GB的权重，从哪加载？

| 来源 | 速度 |
|-----|-----|
| Hugging Face | 慢，受出口带宽限制 |
| S3 | 中等，有网络延迟 |
| 本地缓存 | 快，GB/s级带宽 |

大模型必须用数据中心内的高速存储。

## 路由与负载均衡

多个副本在线，请求发给谁？

**路由器 vs 负载均衡器**

- **路由器**：这个请求应该去哪？（基于请求内容）
- **负载均衡器**：这个请求可以去哪？（基于系统状态）

**智能路由**

不是简单地平均分配。可以基于：

| 策略 | 说明 |
|-----|-----|
| KV缓存感知 | 发给已有相关前缀缓存的副本 |
| LoRA感知 | 发给已加载相应LoRA权重的副本 |
| 序列长度感知 | 长请求发给负载较轻的副本 |

**队列**

流量超过处理能力时，请求进入队列等待。

可以实现优先级队列：付费用户优先、重要请求优先。

## 多云容量管理

P.K.打开了一张全球地图，上面标满了数据中心。

"当你需要几百、几千个GPU时，单一云商可能不够。"

**多云策略**

| 策略 | 说明 |
|-----|-----|
| 主备 | 一个云商为主，另一个备用 |
| 混合 | 根据价格和可用性动态选择 |
| 地理分布 | 不同地区用不同云商 |

**GPU采购**

云GPU紧缺。获取方式：

| 方式 | 特点 |
|-----|-----|
| 按需 | 随时获取，但可能没货 |
| 预留 | 承诺长期使用，保证有货但价格固定 |
| 现货 | 便宜但随时可能被收回 |

大规模部署需要混合策略。

**地理感知负载均衡**

用户在哪，GPU就应该在附近。减少网络延迟。

## 可靠性

"99%的可用性听起来很高，对吧？"P.K.说，"一年大约有3.5天宕机。你的用户能接受吗？"

**可用性级别**

| 级别 | 年宕机时间 |
|-----|----------|
| 99%（两个9） | 3.5天 |
| 99.9%（三个9） | 8.7小时 |
| 99.99%（四个9） | 52分钟 |

闭源API通常是两个9。自建推理可以做到四个9。

**提高可靠性的方法**

1. **冗余**：多副本、多区域、多云
2. **优雅降级**：部分失败时降低服务质量而不是完全停止
3. **快速恢复**：自动检测故障，快速替换失败的副本
4. **混沌工程**：主动注入故障，测试系统韧性

## 零停机部署

更新模型或配置时，不能中断服务。

**滚动更新**

```
v1 v1 v1    ← 当前状态
v1 v1 v2    ← 启动一个新版本
v1 v2 v2    ← 关闭一个旧版本
v2 v2 v2    ← 完成
```

**蓝绿部署**

同时运行新旧两个环境，一次性切换流量。

**金丝雀发布**

先把小部分流量导向新版本，观察一段时间，没问题再逐步扩大。

## 成本估算

"推理不便宜。"P.K.打开一个账单。"你需要知道钱花在哪里。"

**成本组成**

| 项目 | 占比 |
|-----|-----|
| GPU计算 | 最大 |
| 存储 | 中等 |
| 网络传输 | 因场景而异 |
| 其他（日志、监控等） | 小 |

**单位经济**

从"每GPU小时多少钱"换算成：
- 每百万token多少钱
- 每次请求多少钱
- 每个用户每月多少钱

这样才能和产品定价对齐。

**优化成本**

1. 量化减少GPU需求
2. 批处理提高利用率
3. 缓存减少重复计算
4. 现货实例降低单价
5. 缩容空闲副本

## 可观测性

"你不能优化你看不见的东西。"

**三大支柱**

| 支柱 | 作用 |
|-----|-----|
| 指标（Metrics） | 数字化性能指标 |
| 日志（Logs） | 事件记录 |
| 追踪（Traces） | 请求的完整路径 |

**关键指标**

- TTFT（首token时间）
- 每token时间
- 端到端延迟
- 吞吐量（请求/秒，token/秒）
- 错误率
- GPU利用率
- 显存使用

**告警**

设置阈值，异常时自动通知：
- 延迟超过SLA
- 错误率上升
- 副本健康检查失败
- 队列积压

## 档案室

**本章核心概念：**

1. **容器化** = 打包应用和依赖，确保可复现部署。锁定版本，精简镜像，保存好的构建。

2. **自动扩缩容** = 根据流量和利用率动态调整副本数。关键是冷启动速度——越快越敢缩容，省的钱越多。

3. **可靠性 = 冗余 + 优雅降级 + 快速恢复**。自建推理可以超越闭源API的可用性（四个9 vs 两个9）。
