## 秘技一：量化

第一份档案的标题是《精度的艺术》。

**什么是量化**

模型训练时用高精度数字（通常是FP16/BF16，16位）。量化把这些数字转换成低精度格式（FP8、FP4，8位或4位）。

**为什么有效**

1. **预填充更快**：低精度Tensor Core算力翻倍
2. **解码更快**：搬运数据量减半，相当于带宽翻倍
3. **显存更省**：模型占用空间减半

从16位到8位，通常能获得**30-50%的性能提升**。

**风险**

精度降低会引入误差。误差会累积。

考虑圆周率的不同精度：

| 精度 | π² | π³ |
|-----|-----|-----|
| 3.14159 | 9.869588 | 31.006198 |
| 3.14 | 9.8596 | 30.959144 |
| 3 | 9 | 27 |

误差在每次计算中都会放大。

**数字格式**

浮点数有三部分：符号位、指数位、尾数位。

**FP8 E4M3** = 1位符号 + 4位指数 + 3位尾数

指数位提供"动态范围"——能表示很大和很小的数字。这对保持模型质量至关重要。

**微缩放格式**（MXFP8, MXFP4）每32个参数计算一个缩放因子，进一步减少误差。

**量化什么**

不同组件对量化的敏感度不同：

| 组件 | 敏感度 | 建议 |
|-----|-------|-----|
| 线性层权重 | 低 | 可以量化 |
| 激活值 | 低-中 | 可以量化 |
| KV缓存 | 中 | 谨慎量化 |
| 注意力层 | 高 | 通常不量化 |

**生产标准**

量化后要测试模型质量：
- **困惑度（Perplexity）**：越低越好
- **智能基准**：MMLU、SWE-bench等
- **领域评估**：你自己的测试集

目标：质量下降**不可察觉**。

## 秘技二：推测解码

第二份档案封面写着：《用猜测换速度》。

**问题**

LLM解码是自回归的——必须一个token一个token生成。每生成一个token，都要把模型权重从显存搬到计算单元。

计算单元在等数据。闲置的算力被浪费了。

**解决方案**

用闲置的算力**猜测**接下来的几个token，然后一次性验证。

```
传统解码：
  生成token 1 → 生成token 2 → 生成token 3
  （3次完整前向传播）

推测解码：
  猜测token 1,2,3 → 验证全部 → 接受正确的
  （1-2次完整前向传播）
```

**为什么可行**

验证比生成容易得多。就像数独：解一道数独很难，但检查一个答案是否正确很简单。

**核心指标**

1. **草稿成本**：生成猜测需要多少时间
2. **序列长度**：每次猜测多少个token
3. **接受率**：猜对的比例

接受率在序列早期很高，越往后越低。

**主要方法**

| 方法 | 原理 |
|-----|-----|
| Draft-Target | 用小模型生成草稿，大模型验证 |
| Medusa | 给模型加额外的"头"，并行生成多个候选 |
| EAGLE | 用特征向量而非token来预测 |
| N-gram | 基于历史出现过的模式猜测 |
| Lookahead | 并行运行多个解码分支 |

**适用场景**

推测解码在**低批处理量**时最有效。批处理量高时，计算本来就被利用了，推测的收益下降。

## 秘技三：缓存

第三份档案：《KV缓存的秘密》。

**什么是KV缓存**

注意力计算需要查看当前token与所有之前token的关系。如果每次都重算，时间复杂度O(n²)。

KV缓存存储之前token的K和V值。复用它们，时间复杂度变成O(n)。

**前缀缓存**

很多请求有相同的前缀：系统提示、RAG文档、函数签名。

如果能在请求之间共享这些前缀的KV缓存，就能跳过重复计算。

```
请求1: [系统提示] + [用户问题A]
请求2: [系统提示] + [用户问题B]

前缀缓存: 
  [系统提示]的KV值只计算一次，两个请求共享
```

这对TTFT（首token时间）影响巨大。

**缓存存在哪**

| 位置 | 优点 | 缺点 |
|-----|-----|-----|
| GPU显存 | 最快访问 | 空间有限 |
| CPU内存 | 空间大 | 访问较慢 |
| 磁盘 | 容量无限 | 访问最慢 |

**缓存感知路由**

如果你有多个GPU副本，可以根据KV缓存的位置路由请求。

```
请求: "基于文档X，回答..."
路由: → GPU #3 已经有文档X的KV缓存
```

**长上下文处理**

上下文越长，KV缓存越大。128K上下文的KV缓存可能比模型权重还大。

解决方案：
- 量化KV缓存
- 分层存储（热数据在GPU，冷数据在CPU）
- 滑动窗口注意力

## 秘技四：模型并行

第四份档案：《多GPU协同作战》。

**为什么需要并行**

大模型装不进单张GPU。即使装得下，单张GPU也可能不够快。

**张量并行（Tensor Parallelism）**

把模型的每一层切分到多张GPU上。

```
Layer 1: [GPU 0 部分] + [GPU 1 部分] + [GPU 2 部分]
Layer 2: [GPU 0 部分] + [GPU 1 部分] + [GPU 2 部分]
...
```

优点：**降低延迟**。每个GPU只计算一部分，然后同步。

缺点：需要频繁的GPU间通信。NVLink带宽是瓶颈。

**流水线并行（Pipeline Parallelism）**

把模型的不同层分配到不同GPU。

```
GPU 0: Layer 1-20
GPU 1: Layer 21-40
GPU 2: Layer 41-60
```

优点：GPU间通信少。

缺点：有"气泡"——当GPU 0在算第二个请求时，GPU 1在等。

**专家并行（Expert Parallelism）**

专门用于MoE模型。把不同的专家放在不同GPU上。

```
GPU 0: Expert 1-32
GPU 1: Expert 33-64
...
```

每个token只激活部分专家，所以只有部分GPU需要工作。

优点：**高吞吐量**。

缺点：需要All-to-All通信，路由不均衡会导致负载不平衡。

**多节点推理**

当模型大到单台服务器装不下（如万亿参数），需要跨节点推理。

用InfiniBand网络连接节点，但网络延迟比NVLink高得多。

## 秘技五：分离

最后一份档案：《预填充与解码的分离》。

**问题**

预填充和解码有不同的瓶颈：
- 预填充：计算受限
- 解码：内存受限

但传统架构把它们放在同一个GPU上。长输入的预填充会阻塞其他请求的解码。

**解决方案：分离（Disaggregation）**

把预填充和解码分开，在不同的GPU（池）上运行。

```
请求 → 预填充GPU池 → KV缓存传输 → 解码GPU池 → 输出

预填充GPU：优化计算
解码GPU：优化内存带宽
```

**关键挑战**

KV缓存必须从预填充GPU传输到解码GPU。这需要高速网络。

解决方案：
- 量化KV缓存减小传输量
- 使用NVLink或高速网络
- 智能调度，批量传输

**什么时候用分离**

- 输入很长（RAG、文档处理）
- 批处理量大
- 有足够的流量来填满两个池

**动态分离**

NVIDIA Dynamo支持动态分离——根据当前负载自动调整预填充和解码的GPU比例。

## 技术之间的关系

P.K.在白板上画了一张图。

"这些技术不是独立的。它们互相影响。"

**协同**
- 量化KV缓存 + 分离 = KV传输更快
- 量化 + 缓存 = 能存更多缓存
- 张量并行 + 推测解码 = 延迟更低

**冲突**
- 高批处理 + 推测解码 = 推测收益下降（计算已饱和）
- 过度量化 + 长上下文 = 质量下降（误差累积）

**推理工程师的目标**

> *"创造一组平衡的优化，让整体效果大于各部分之和。"*

## 档案室

**本章核心概念：**

1. **量化** = 降低数字精度换取速度。FP8是目前的甜点，质量损失可控。注意力层通常不量化。

2. **推测解码** = 用闲置算力猜测多个token，一次验证。适合低批处理量场景，能显著降低延迟。

3. **分离** = 预填充和解码在不同GPU上运行，各自针对瓶颈优化。需要高速KV缓存传输。
