## CUDA：一切的基础

**CUDA不是编程语言**

CUDA是NVIDIA的GPU编程平台。你用C++写代码，然后CUDA编译器把它编译成GPU能执行的指令。

**内核（Kernel）**

内核就是"一段在GPU上运行的并行代码"。

最简单的例子：把一个数组的每个元素乘以2。

```cpp
__global__ void double_array(float *arr, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        arr[idx] = arr[idx] * 2;
    }
}
```

在CPU上，这个操作是线性的——一个一个处理。  
在GPU上，数千个元素可以同时处理。

**为什么内核重要**

传统的注意力算法可以用几十行代码表达。

但**FlashAttention**——同样的数学运算——需要数万行代码来实现更高效的内存访问模式。

算法和实现是两回事。推理优化本质上是优化实现。

## 内核选择与内核融合

P.K.打开了一个代码库。

"大多数推理工程师不需要自己写内核。但你需要知道如何选择内核。"

**现成的内核库**

- **cuBLAS**：CUDA版的基础线性代数库，GEMM（矩阵乘法）是最常用的
- **cuDNN**：神经网络原语
- **CUTLASS**：高性能内核的模板库
- **FlashInfer**：LLM推理专用的优化内核

**内核选择**

内核是为特定硬件写的。H100的内核可能无法在B200上发挥最佳性能，反之亦然。

大部分内核选择是自动的——框架和引擎会根据硬件选择预配置的内核。

但对于关键算法，手动选择更好的内核可以显著加速推理。比如DeepSeek发布的**DeepGEMM**内核，专门优化了FP8矩阵乘法。

**内核融合**

假设有两个连续的操作：乘以2，再乘以3。

朴素执行：
1. 从内存读数据
2. 乘以2
3. 写回内存
4. 从内存读数据（同样的数据！）
5. 乘以3
6. 写回内存

步骤3和4是浪费——数据从内存到计算单元走了个来回。

**内核融合**把两个内核合并成一个：直接乘以6。读一次，写一次。

在解码阶段（内存瓶颈），这种优化至关重要。

## PyTorch：行业标准

"如果你只能学一个工具，就学PyTorch。"P.K.说。

**什么是PyTorch**

一个Python包，用于描述张量运算。最初由Meta创建，现在属于Linux基金会。

它是训练和推理的行业标准。

**为什么PyTorch强大**

1. **用Python写高性能代码**：不需要C++也能跑GPU
2. **可以插入CUDA内核**：需要精细控制时随时可以降级
3. **自动梯度计算**：训练必备，推理也有用
4. **丰富的生态**：从模型定义到部署的完整工具链

**torch.compile**

把PyTorch模型编译成针对特定GPU优化的代码。自动进行内核选择和内核融合。

但对于LLM推理，大部分内核是定制的（比如FlashAttention），`torch.compile`无法融合它们。所以LLM推理通常直接用推理引擎。

## 模型文件格式

P.K.指向一堆文件。

"模型权重怎么存储？"

**safetensors**

Hugging Face创建的格式。专门用来存储张量数据。

- **安全**：不能执行任意代码（不像通用的pickle格式）
- **高效**：使用内存映射，加载大文件不会爆内存
- **标准**：几乎所有开源模型都用这个格式

大模型的权重有几百GB，分成几十个safetensors文件。

**ONNX**

开放神经网络交换格式。不只存权重，还存执行图。

优点：高度可移植，跨框架、跨硬件。  
缺点：导出过程可能有兼容性问题。

## 推理引擎：vLLM与SGLang

"这是你日常工作最多的一层。"P.K.打开了一个vLLM的部署界面。

**什么是推理引擎**

基于PyTorch的高度优化的推理服务。支持常见模型架构，开箱即用。

三大主流引擎：

| 引擎 | 维护者 | 特点 |
|-----|-------|-----|
| vLLM | Linux基金会 | 最流行，支持广泛，开源 |
| SGLang | LMSYS | 性能强劲，结构化输出 |
| TensorRT-LLM | NVIDIA | NVIDIA专属优化 |

**vLLM的核心优化**

1. **PagedAttention**：像操作系统管理内存一样管理KV缓存
2. **连续批处理**：动态合并请求，最大化吞吐量
3. **自动内核选择**：根据模型和硬件选择最优内核

**SGLang的特点**

1. **RadixAttention**：更高效的前缀缓存
2. **原生结构化输出**：JSON schema约束
3. **编程接口**：可以编排复杂的多轮推理

**如何选择**

大多数情况下，vLLM是默认选择——社区大、文档全、兼容性好。

SGLang在某些场景（长前缀、结构化输出、复杂流水线）可能更快。

建议：在你的模型和工作负载上都测试一下。

## TensorRT-LLM

"NVIDIA自己的推理引擎，值得单独说。"P.K.打开另一个终端。

**特点**

- NVIDIA专属优化，和硬件集成最深
- 需要把模型编译成TensorRT引擎格式
- 支持C++ API，适合低延迟场景
- 某些场景下性能最好

**缺点**

- 只支持NVIDIA GPU
- 模型兼容性不如vLLM广泛
- 编译过程可能需要调试

**什么时候用**

- 你只用NVIDIA GPU
- 你需要极致性能
- 你愿意投入时间调优

## NVIDIA Dynamo

P.K.在白板上画了一个更大的架构图。

"当你有几百、几千个GPU时，单机推理引擎不够了。你需要编排。"

**Dynamo是什么**

NVIDIA的分布式推理编排系统。坐在推理引擎之上，管理：

- 多GPU、多节点的模型分片
- 请求路由和负载均衡
- 自动扩缩容
- 预填充/解码分离

**关键功能**

1. **分离（Disaggregation）**：预填充和解码可以在不同的GPU上运行
2. **动态路由**：根据KV缓存位置路由请求
3. **专家并行**：MoE模型的高效推理

**什么时候用**

- 大规模部署（100+ GPU）
- 需要预填充/解码分离
- 运行MoE模型

## 性能基准测试

P.K.递给我一份测试报告。

"软件的好坏要用数据说话。"

**测试什么**

- **延迟**：TTFT、每token时间、端到端
- **吞吐量**：每秒请求数、每秒token数
- **饱和点**：多少并发请求会让系统过载

**测试工具**

- **genai-perf**：NVIDIA的基准测试工具
- **guidellm**：社区工具
- **自定义脚本**：针对你的具体工作负载

**测试陷阱**

1. **输入输出长度影响巨大**：要用真实的长度分布
2. **预热很重要**：前几个请求可能很慢，不代表真实性能
3. **别只看平均值**：看P50/P95/P99
4. **复现环境**：确保测试环境和生产环境一致

## 档案室

**本章核心概念：**

1. **软件栈层次** = CUDA（GPU编程）→ PyTorch（深度学习框架）→ 推理引擎（vLLM/SGLang）→ Dynamo（分布式编排）。越往上越抽象，越往下越精细。

2. **内核融合** = 把多个操作合并成一个内核，减少内存读写。解码阶段的关键优化。FlashAttention就是融合后的注意力内核。

3. **推理引擎选择** = vLLM是默认选择，SGLang在特定场景可能更快，TensorRT-LLM提供NVIDIA专属优化。在你的工作负载上实际测试。
