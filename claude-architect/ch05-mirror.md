# 镜厅国 · 迷途与归途

> 这章要解决什么问题？
> 为什么 agent 在长对话里会"忘"？"lost in the middle" 是什么？Escalation 什么时候触发？Provenance 怎么保住？

---

## 一

镜厅国是五个国家里最难走的一个。

首都"**上下文回廊**"是一条被镜子围起来的走廊——每一面镜子都反射前一面的画面, 镜子越多, **中间那几面上的画像越模糊**.

旅行者走了一段才明白:

> **这就是 lost-in-the-middle**.
>
> Agent 多轮对话后, 最早的 turn 和最新的 turn 记得清, **中间几轮 LLM 的注意力最低**——**重要信息放在中间就消失**.

向导解释:"LLM 的 attention 对'首尾 prominent, 中间 dampened'有经验偏置. 200 页文档塞给它, **它记得头 30 页和尾 10 页, 中间 160 页模糊如雾**."

旅行者在回廊中抄下来:

> **Context 里越早/越晚的信息保留率越高, 中间段最易丢.**
> 设计提示词时: 关键信息放开头或结尾, 别埋中段.

---

## 二

镜厅国有一个**渐进总结陷阱**的警告牌——

"你可能会想: '中间忘了就总结呗, 让 agent 每几轮做一次总结'."

"看上去合理."

"**实际上**:"

牌子下面列了具体的败笔——

- 总结时 LLM **把细节丢了**: "用户提到 5 个要求" → 总结成"用户有几点需求" → 再下一轮就真的不知道是 5 个了
- 总结时 LLM **做因果重构**: "用户说 A 和 B" → 总结成"用户说 A 因为 B" → 加了原文没有的因果
- 总结时 LLM **重新加权**: 用户轻描淡写的一句话可能被总结放大, 重要的一段话被压缩

"**渐进总结 = 每一轮都丢一点信息, 最后全是影子**."

"那怎么办?"

"**保留 claim-source mapping**——谁说的什么, 什么时候说的. Provenance."

## 三

走廊的中段, 旅行者遇到了**Escalation 决策亭**.

亭里有个亭长——一个戴眼镜的老人——专门判断:**agent 什么时候该停下来, 交给人?**

"三种情况." 亭长说.

1. **政策空白**: 用户的要求落在 agent 没被授权决定的地带. 比如"能否给我退 1 万美元?"——agent 没有这个权限.
2. **客户明确要求**: 用户说"我要找人工". 不 escalate 就是忽视用户意愿.
3. **agent 自己推不下去了**: 尝试了所有可行路径, 还是卡住. 继续只会浪费轮次.

"不 escalate 是常见反模式." 亭长说. "很多人训练 agent '尽量别 escalate', 结果 agent 会**胡诌**——假装有权限, 假装能做, 最后越帮越乱."

旅行者在笔记上记:

> **Escalation 不是失败, 是 calibrated failure.**
> 不知道边界的 agent, 伪装成知道——才是失败.

## 四

镜厅国的最后一节课是**多实例评审**.

旅行者走到回廊的尽头, 看到一个小房间, 房间里有两个 agent——A 和 B.

A 刚做完一个任务. B 被带进来, **没看过 A 之前的推理过程**, 只看**最终产出**.

"A 的 context 被擦掉了. B 是全新的 context 来看这份成果."

"为什么这么麻烦?"

"因为**自审有偏见**." 管理员说. "A 审查自己刚做的, 它会**合理化**. B 是全新视角, 不被前因污染."

这让旅行者想起洞窟国的**独立审查实例**——是同一个思想. 多实例评审就是**用'新的 context 容器'来打破'自我合理化'偏见**.

---

## 五

他在镜厅国待到最后一天, 写了一段总结:

> **镜厅国教的是 "agent 在长距离下的可靠性"**.
>
> - Lost-in-the-middle → 重要信息放位置
> - 渐进总结陷阱 → 不要无脑总结, 要**保 provenance**
> - Escalation → 知道边界比装懂更高级
> - 多实例评审 → 自审 ≠ 独立审
>
> 一句话: **agent 在 100 轮后还能对的原因, 永远来自工程 discipline, 不来自模型变聪明**.

---

## 关键概念

- **Lost-in-the-middle**：LLM 对 context 中段注意力最低。重要信息应放开头或结尾。
- **渐进总结陷阱**：多轮总结会逐轮丢信息 + 加因果重构 + 重新加权。
- **Provenance / Claim-source mapping**：每个结论对应到它的原始信息源。总结时必保。
- **Escalation 决策**：政策空白 / 客户明确要求 / agent 推不下去 — 这三种必须停手找人。
- **多实例评审**：用独立 context 的 agent 去审第一个 agent 的产出，避免自审偏见。

## 我的理解

- **Agent 可靠性**主要不是模型问题, 是**工程 discipline 问题**.
- **总结**不是"节省上下文", 是"损失可追踪性"的代价交换——很多场景下得不偿失.
- **Escalation 是美德**. 训练 agent "永远别 escalate" 是反向训练.

## 类比

- Lost-in-the-middle 像**一个人听长篇会议**: 开头和结尾的议题记得, 中段的具体讨论经常模糊.
- 渐进总结陷阱像**复印机复印复印件**: 每一次复印都丢一点, 多次之后完全不像原件.
- Escalation 像**医生的 "我不知道"**: 一个医生愿意说"这不是我的专业"比"什么都能治"更可信.
- 多实例评审像**庄家 + 公证人**: 庄家说这是牛肉, 第二个人(不知道庄家怎么做的)来独立判断是不是——才有可信度.

## 遗留问题

- Provenance 的实现代价——每个 claim 存一个 source pointer, context 会不会膨胀? 可能需要"sparse provenance"(只对关键 claim 存)?
- Escalation 的触发"agent 自己推不下去"怎么自动化检测? 用**连续 N 轮无新进展**还是**tool_error 率**? 场景题可能问.
