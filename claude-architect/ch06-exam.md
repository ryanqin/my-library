# 终章 · 考场上的那六道场景题

> 这章要解决什么问题？
> 六道场景题怎么解？**跨 domain 的真实 bug** 长什么样？走过五国的人如何应考?

---

## 一

考场设在五国交界的 "Foundations 厅". 他走进去, 看见桌上一张单子, 写着六道场景题——

每一道都不是"请问 X 是什么", **都是"这里出了问题, 你怎么修"**.

他深吸一口气, 开始读.

---

## 二 · 场景 1

> 某 agent 被用来做合规审批. Prompt 里写"**务必**在任何付款前先查合规政策". 上线 1 个月, 合规团队发现 85% 的付款前有查, 15% 没查. 你怎么修?

他笑了一下——这是**编排国**的反模式. 一秒内写下答案:

> **用 Hook 替代 prompt 强制**. `on_tool_use(name="make_payment")` 前强制调 `check_policy`, 没调就拒绝. Prompt 是建议, Hook 是法律.

---

## 三 · 场景 2

> 某 agent 有 50 个 tool. 用户说"帮我 debug 这个 Python 文件", agent 总是选不到 `python_test` 这个 tool, 反而选了 `bash_execute`. 你怎么修?

**工具国** 的问题. 答案跳到笔尖:

> `python_test` 的 description 不够明确. 写成 "用户提到 debug、测试、pytest、Python 代码质量时使用". description 决定选择率, 不是 schema 也不是名字.

---

## 四 · 场景 3

> Agent 在一个 monorepo 里跑. 用户反馈: agent 在 `backend/` 里用 Python 回答, 但进到 `frontend/` 里还坚持用 Python. 你怎么修?

**洞窟国** 的分层题.

> `frontend/CLAUDE.md` 里没有声明语言. 加一行 `Use TypeScript here`. CLAUDE.md 是**叠加**的——全局+项目+目录, 最具体的层赢. Dir 级 CLAUDE.md 可以覆盖项目级的默认.

---

## 五 · 场景 4

> 需要 agent 产出一份结构化合同摘要. 现在每次产出的字段顺序和命名都不稳定. 你怎么修?

**结构国** 的核心.

> 三步:
> 1. Few-shot — 给 3-5 个样例摘要 (最有效)
> 2. `tool_use` 强制 JSON — 通过 save_summary tool 的 input schema 消除语法错
> 3. Schema validator 兜底 — 语义错靠后置校验

---

## 六 · 场景 5

> 客户 support agent 被设计成"尽量自己解决, 别打扰人工". 运行 3 个月后, 客户投诉率激增, 分析发现 agent 经常**假装能解决实际解决不了的问题**. 你怎么修?

**镜厅国** 的 escalation 课.

> Agent 训练成"尽量别 escalate"是反向. 改成:
>
> Escalation 触发条件:
> 1. 政策空白(agent 没被授权决定的问题)
> 2. 客户明确要求人工
> 3. Agent 连续 N 轮无推进
>
> 不 escalate 不是美德, **calibrated escalation 才是**.

---

## 七 · 场景 6(跨 domain)

> Agent 跑一个 50 轮的研究任务. 第 40 轮时, agent 引用了一个"用户在第 5 轮提到的约束", 但实际上用户第 5 轮根本没这么说. 你怎么修?

他停住了笔.

这道题不属于单一国家. 它同时涉及:

- **编排国**: 长任务要 coordinator-subagent 分 context
- **镜厅国**: lost-in-the-middle + 渐进总结陷阱
- **镜厅国**: Provenance — 每个 claim 要有 source pointer

他的答案一口气写下来:

> 这是**渐进总结丢信息 + provenance 缺失**的复合 bug.
>
> **解法叠加 3 层**:
>
> 1. **Claim-source mapping**: 每个 agent 引用的用户约束必须带上"turn-N 的原文片段"作为 source. 不能引用没有 source 的约束.
>
> 2. **避免渐进总结**: 不要用"每 10 轮做一次总结"的策略. 要么保留全原文 (context 够的话), 要么用 **coordinator-subagent** 把子任务拆到独立 context 容器.
>
> 3. **多实例评审**: 跑完后让一个新 agent (全新 context) 审查"引用的约束是否都有出处". 自审捕不到这种 bug.
>
> 根本原因: **"agent 记错"不是模型能力问题, 是工程架构问题.**

---

## 八

他抬起头. 考场里其他考生还在写.

窗外天色已晚. 他想起五夜的旅行——编排国的王宫, 工具国的集市, 洞窟国的阶梯, 结构国的匠人, 镜厅国的回廊. 每一个场景都曾是真实世界里 agent 崩溃的记录.

他把考卷交上去, 走出考场. 门口, 他看见五国的国王站成一排——每人手里拿着一束自己国度的光.

——他 一下 站住.

蓝、金、土、白、彩色 碎片.

**这 是 第一夜 的 那 个 梦**.

那 一 夜 他 还 没 出发. 五 个 国王 围着 他 问 同 一个 问题: "如果 只能 选 一件 事 带回 去, 你 选 什么?" 他 当时 答 不出 来.

现在 五个 国王 真 的 站 在 这 里, 每人 一束 光. 他们 没有 再 问 那个 问题——**他们 在 等 他 把 答案 说 出来**.

他 看 着 五束 光. 他 知道 现在 他 答 得 出 来 了:

"——**一束 都 不 选**."他 在 心里 说,"五束 必须 一起 带 回去. 编排 的 蓝、工具 的 金、洞窟 的 土、结构 的 白、镜厅 的 彩——少 一束, agent 就 立 不 住. 我 不 是 来 选 一个 国家 的, 我 是 来 把 五种 光 都 带回 真 实 世界 的."

五个国王同时对他鞠了一躬. 五束光在他面前合成一道——**那道光 和 真实 的 agent 行为 长 得 一模一样**.

旅行者 也 鞠 了 一 躬.

"五国游记, 到此结束." 他在心里说. "但 agent 的工作, 才刚刚开始."

---

## 关键概念 · 回顾

**五国地图:**

| 国 | 占比 | 核心反模式 | 正确姿势 |
|---|------|----------|---------|
| 编排 | 27% | 用 prompt 强制业务规则 | 用 Hook |
| 工具 | 18% | tool description 含糊 | 明确触发语义 |
| Claude Code | 20% | 所有配置塞一个文件 | CLAUDE.md 三层叠加 |
| 结构化 | 20% | 只给 schema 不给样例 | Few-shot + tool_use |
| 上下文 | 15% | 渐进总结 + 不 escalate | Provenance + 多实例评审 |

## 我的理解

- 考试题**不考概念**, **考反模式**.
- 真实 bug **跨 domain**. 五国训练的价值就是——碰到复合 bug 时能拆开看.
- 最难的题总是**"看上去合理但在长距离下崩溃"**——那都在镜厅国的范畴.
