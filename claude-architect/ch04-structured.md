# 结构国 · Few-shot 的匠人街

> 这章要解决什么问题？
> 怎么让 LLM 产出**格式稳定**的东西？Few-shot 为什么比 schema 还有效？`tool_use` 强制 JSON 又能解决什么？

---

## 一

结构国是个由匠人组成的国家。

首都 Few-shot 街上，每一个匠人店铺的橱窗里都摆着**样品**——不是说明书，不是规格表，**是已经做好的成品**。

旅行者走进一家:"我要定做 3 个同款的茶杯." 匠人没说话, 把橱窗里的样品茶杯递给他看——"就这样的."

旅行者明白了:"要复制样品? 不是写要求?"

"**写要求**让人以为自己听懂了, 做出来还是歪的." 匠人说. "**给样品**才能避免任何歧义——你 DIY 产出的每一件都会和样品长一样."

旅行者掏出笔记:

> **Few-shot = 给样品. 不是描述, 是例子.**
>
> Schema 说"要输出 {name, age, email}"
> Few-shot 说"上次我是这么输出的: {name: 'Bob', age: 30, email: 'bob@x.com'}"
>
> 第二种让 LLM 的产出稳定 10 倍.

这一课简单得让他想笑, 但他知道它是结构国的**根本教义**——**模糊场景下的最强对齐手段**.

---

## 二

街的第二段, 一家店挂着招牌——"**JSON 专门店**"。

店里的师傅演示了一个戏法——他让 LLM 产出 JSON, 但**根本不提 "JSON" 这个词**.

"怎么做到的?"

师傅拿出一把锁和钥匙的图:

```
tool_use(name="save_result", input={"field1": ..., "field2": ...})
```

"**我不叫它输出 JSON, 我叫它用 save_result 这个 tool.**"

"LLM 知道 tool 的 input 必须是 JSON——**它会强行把自己的想法挤进 JSON schema**. 这就消除了'LLM 语法错误'这种最恶心的 bug."

"但是——" 师傅举起一根手指. "这只消除**语法错**. **语义错**还在——LLM 可能会把用户的年龄写在 email 字段里. tool_use 救不了."

旅行者抄下这一条:

> **tool_use 强制 JSON = 消除语法错, 不消除语义错.**
>
> 语义错还是要靠 Few-shot + 校验.

---

## 三

街的尽头, 旅行者看到一家**批发商**——**Message Batches API**.

店里正在打折销售:

> **50% off!** — 所有请求打对折
> 条件: 24 小时内完成, **不支持多轮 tool calling**

旅行者问店主:"什么时候用这个?"

店主掰着手指算:
- 一次性的大批量处理(比如给 10000 封邮件做分类) — 用
- 实时对话 — 不能用(24 小时是宽松 SLA, 不是秒级)
- 需要多轮 tool_use 的复杂 agent — 不能用(不支持)

"一句话: **离线批处理, 对延迟不敏感, 不需要 agent loop**."

---

## 四

结构国的最后一晚, 他在日记本上画了一张图:

```
要格式稳定 → Few-shot 样例 (最强)
          ↓
要消除语法错 → tool_use 强制 JSON
            ↓
要消除语义错 → 再来一轮 Few-shot + 校验

离线批量 + 宽 SLA → Message Batches API (50% off)
实时 + 多轮 → 标准 API
```

**结构国教我的不是"怎么写 schema", 是"怎么让输出稳定"**. 稳定不等于严格——稳定是"十次请求十次长得一样".

---

## 关键概念

- **Few-shot**：给样例 > 给规格。**模糊场景下最强的对齐手段**。
- **`tool_use` 强制 JSON**：用 tool 的 input schema 间接逼 LLM 产 JSON。消除语法错，**不消除语义错**。
- **Message Batches API**：50% 折扣，24h 完成，**不支持多轮 tool calling**。离线批量场景。

## 我的理解

- **稳定 ≠ 严格**。稳定是可重复性, 严格是零容错——两者手段不同.
- **Few-shot > Schema** 是反直觉的——工程师会本能信 schema, 但 LLM 不是解释器, 它看例子比看规则学得快.
- **tool_use 强制 JSON** 不是"让 LLM 用 tool", 是"借 tool 的 schema 作为结构化输出的承载"——tool 只是借壳上市.

## 类比

- Few-shot 像**教小孩洗碗**: 你不跟他讲"洗碗要经过清洁/冲洗/晾干三个阶段", 你**做给他看**, 他第二天自己就会.
- tool_use 强制 JSON 像**给犯人一张表格**: 空白页他会写散文, 有表格他只会填格子.
- Message Batches API 像**快递的经济型服务**: 48 小时到, 便宜一半——但你急件就别选了.

## 遗留问题

- Few-shot 几个 shot 最优? 我感觉 3-5 个, 但考试可能给具体数字题.
- tool_use 强制 JSON 的陷阱: 如果 tool 本身不应该被调用(业务上),但你为了 JSON 硬 call 了一个"假 tool", 合适吗? 我觉得不合适——应该用 Anthropic 的 "structured output" 专门接口.
