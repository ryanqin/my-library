# 第三夜 · 一楼的发动机和铜尺

> 这章要讲什么?
> 沈姨带阿衍看一楼前台后面那间小屋. 一台旧笔记本, 一把铜尺. llama-cpp-python + GGUF + sentencepiece —— 发动机和量字数的那把尺.

---

## 一

第三天晚上.关店之后.

沈姨没让阿衍下地下室,而是把柜台后面一道隔板拉开.

阿衍这才看见,前台后面其实是一间**很小的工作间**,大概三坪.里面没有装修,水泥地,一张木桌,桌上有一台五年前的笔记本电脑,机身有点偏旧,但风扇吹得很匀.桌侧立着一个塔扇,稳稳地朝笔记本吹.

笔记本旁边——

阿衍多看了两眼.

那里放着一把**铜尺**.手工的,长不到二十公分,边角磨亮了,中间刻着小数点和不规则的刻度.

---

## 二

"沈姨,那是?"

"那是我量字数的尺."

"——量字数?"

"对.客人那段话,我得知道'有多重'再决定怎么处理."

她把笔记本竖起来,屏幕亮起一行小字:

```
$ llama-cpp-server --model gemma-4-e2b-it-Q4_K_M.gguf --n-ctx 8192
[loaded]  context = 8192   gpu_layers = -1   metal = on
```

"这是发动机."她拍了拍笔记本外壳."这台机器只有一个职责——**读一个 GGUF 文件,吐 token**.整栋楼里只有这间小屋知道'llama.cpp'这四个字."

"上面所有的楼层都不知道?"

"都不知道.它们都通过一根管道把请求塞进来,再从管道接住答复.上面如果哪一天我换发动机——比如把这台旧本本换成手机,或者换成 Mac mini——**只要这间屋子的接口不变,楼上的代码一行都不用改**."

阿衍记下:

> **L1 Runtime Abstraction. 整栋楼的 import-llama-cpp 只在这一间屋.**

---

## 三

"那个 GGUF 文件——"阿衍指着屏幕上的路径,"——里面装了什么?"

沈姨拉过一把椅子,让阿衍坐下.

"GGUF 是一个**自带一切**的盒子.里面有压过的权重,有架构信息,有它自己用的 tokenizer,还内嵌了一段对话模板.所以**整个文件,你拷给别人,别人不用装别的库,就能跑**."

她打开屏幕上的 finder,文件夹里有三个 gguf:

```
gemma-4-e2b-it-Q4_K_M.gguf    ← 部署用,中档机能跑
gemma-4-e2b-it-Q8_0.gguf      ← 开发用,质量更接近原版
gemma-4-e4b-it-Q4_K_M.gguf    ← 高挡位,处理图像/语音/复杂推理
```

"为什么是 Q4?"

"Q4 是把权重压到大概一个字节四个 bit.压得越狠,文件越小、跑得越快,但精度会掉一点.Q4_K_M 这一档是社区共识的'**够用且省**'.我开发的时候在另一个房间里用 Q8 对照——但客人面前的这台机器,是 Q4."

阿衍记下:

> **Q4_K_M = 中档机部署主流档. 开发机用 Q8 比对效果.**

---

## 四

"陷阱."沈姨说,"我这台机器我已经踩过四次坑了,我都告诉你."

她伸出右手,扳着手指数:

"**第一**, chat_format 这个东西不能瞎选.你装这个库,它会问你'我用什么对话模板'.每个模型自己有自己的一套口音.Gemma 你给它别人的模板,它会胡言乱语."

"那要怎么办?"

"两条:或者明确选'gemma',或者**让它自己读 GGUF 里面内嵌的模板**——别填.GGUF 文件已经把对话模板背在身上了."

"**第二**, tool-call 这件事——我下次再给你细讲——但你先记住一个事实:**模型回话的 tool_call 不会规规矩矩地放在 message.tool_calls 字段里**.它经常**藏在 content 字段里**,以特殊 token 包着.你要自己写一个 parser 把它捞出来.这是这间小屋里**最难的八十行代码**."

阿衍认真地把这一条写到本子最显眼的位置:

> ⚠ **tool_call 解析自己写. 不能依赖库吐结构化字段.**

"**第三**, n_ctx——也就是上下文窗口——别一上来就开 128K.开越大,KV cache 占内存越大.我这台机器开 8K 已经吃 800MB.开 128K 直接十几个 G,机器爆了."

"那 128K 不是 Gemma 的卖点吗?"

"卖点.但你不一定每天都吃这个卖点.每天你的客人就那几句话——8K 足够.偶尔守夜人晚上做长文聚类,我才把 ctx 临时撑大."

"**第四**——"沈姨拿起桌上那把铜尺,"——'量错字数'."

---

## 五

她把铜尺平放在桌上,招手让阿衍凑近.

"客人那段话,有多少 token?如果你不量,只是数英文里有几个空格、中文有几个汉字——你会**算错很多**."

她翻开笔记本,在终端里敲了一行:

```python
>>> sp.encode("你好世界")
[235256, 235256, 235256]
>>> len(sp.encode("you'll grab subjunctive tomorrow"))
11
```

"看见没.一个四个字的中文是 3 个 token, 一句五词的英文是 11 个 token.差远了."

她拿起铜尺.

"这把尺的作用,就是**用模型自己的方法**去量长度.它叫 sentencepiece.整间小屋里,我有两个尺——一个内嵌在发动机里、一个独立摆在桌上.独立这把,是为了——"

她伸出三根手指数:

```
1. 守夜人在后台预算每日总量时, 发动机不一定加载
2. Android 客户端要算 token 数, 不应该在客户端塞个 GGUF
3. 单元测试里 MockRuntime 没权重, 也要量
```

"——这三种场合,我用独立的 sentencepiece, 不依赖发动机."

阿衍记下:

> **量错 token = orchestrator 预算穿透 = context 被截 = 模型胡说. 用真尺.**

---

## 六

"为什么不用 word-split?"沈姨问回他.

"因为中文没空格, 英文里 'you'll' 是两个 token——不是一个词.切空格切错."

"对.我以前一个朋友造的另一栋楼,就是用 word-split 估的 token 数,结果他半夜接到客户电话说'你这个 agent 怎么动不动就把我前面说过的话忘了'——他算的预算是 4000,实际上已经塞了 5500——超出 context, 就被砍."

"砍掉的是哪部分?"

"通常是**最早**那部分.也就是客人最早说的那句最重要的话.所以——

**算错 token, 不是节省了 token, 是丢了最关键的那一句.**

写在本子上,**第一行**."

---

## 七

阿衍把这一句写在他笔记本的扉页.

外面又起风.红灯笼在窗外晃.沈姨把屏幕合上,塔扇也关.她拿起那把铜尺,轻轻往桌沿一磕——尺尾发出"叮"的一声短脆的响.

"这把尺,是你今天该带回客房的东西.我送你的——不是这把铜的,是它代表的**'真量真算'**."

她看了一眼天花板.

"明天我们上二楼.二楼是这家店里**最迷的一层**——因为它是个'楼中楼'.里面有六层小楼,每一层都有一种活."

她朝楼梯口走了几步,回过头:

"你今晚回去再把这一夜过一遍.记住一句话——

**'楼上不该知道楼下的库叫什么名字.'**"

阿衍把铜尺的样子在笔记本里画了一遍.很丑,但很安心——他知道,以后他写每一行 token-counting 的代码时,都会想起这把尺.
