# 序章 · 翻译者的义务

> 这章要解决什么问题？
> 为什么要把一个 TypeScript agent harness**逐字翻译**成 Python？这是"移植"，还是一种更古老的活——**考古**？

---

## 一

凌晨一点，他终于从终端里退出来。屏幕上还亮着一行 `claw-code --help`，像个刚被唤醒的灯塔。

他原本只打算读一下源码。TypeScript 的那套 harness 最近在 Hacker News 上被讨论得厉害——Claude Code 的执行骨架，带 20 多个子命令，会分词打分，会做惰性加载。他想看看人家怎么写。读了三页，他合上了书：
"这东西得自己写一遍才懂。"

但"自己写一遍"有两种方法。第一种是重新发明轮子——换个语言、换个哲学、从零开始想一遍。第二种是**逐字翻译**。

他选了第二种。

---

## 二

翻译比重写难。重写的时候，任何细节都可以借口说"那不是我的选择"；翻译要你在每一行前停下来问：

> **作者当时为什么这样写？**

他写下三条翻译者的义务：

1. **保留结构**。子目录、命名、分层都照搬。只要 TS 的 `runtime.ts` 在那里，Python 的 `runtime.py` 就在同一个相对位置。
2. **保留边界**。TS 里用 frozen dataclass 的地方，Python 就用 `@dataclass(frozen=True)`。TS 里的 JSON 快照，Python 就继续从 JSON 加载——即使 Python 更喜欢直接写类。
3. **不改行为**。遇到看不懂的细节，**先翻译，再理解**。强行理解会让你先入为主，先翻译才能看见原文隐藏的决策。

这三条让他放心了。他像一个旧图书馆里的手抄僧——不问这段话为什么这样写，先抄下来。

---

## 三

屏幕左边是原作者的 TypeScript。
屏幕右边是他的 Python 空白文件。

他把光标停在 `runtime.ts` 的第一行，深呼吸，开始敲第一个字：

```python
# runtime.py — 翻译自 Claude Code 的 runtime.ts
# 逐字对照，不做改动。不理解的地方留 # TODO: 为什么？
```

他知道这本书会写五夜。

- **第一夜**是大门：`main.py` 那个 argparse 分派器。
- **第二夜**是心脏：`runtime.py` 里的 token 路由和会话管理。
- **第三夜**是记忆：`tools.py` 的 JSON 快照和多层过滤。
- **第四夜**是守门：`permissions.py` 那份沉默的黑名单。
- **第五夜**是遗憾：`parity_audit.py` 那张**未完成的对等地图**。

他保存文件，关掉编辑器。明天早上他要上班，但**今晚开始，他在书桌上有了一份第二工作**——翻译。

---

## 关键概念（边走边记）

- **Harness**：把 LLM 的 tool-use 循环用代码包起来，变成"能跑命令的 agent"的那层执行骨架。
- **逐字移植 (parity port)**：不优化、不重构、不改命名；最大限度保留原作者的意图。
- **frozen dataclass**：不可变数据类——改一点就得构造一个新实例。是 Claude Code 原作里最突兀也最重要的选择之一。

## 我的理解

翻译 = 强迫自己**以作者的抽象粒度**思考。重写会让你逃到自己舒服的粒度去，那就读不到原作的 80% 了。

## 类比

像一个西方哲学专业的中国学生要读懂康德，不能上来就"用自己的话总结"——那会把康德变成你自己。你得先**一字一字抄**，体会德文的句法卡住了你多少次——那些地方就是康德在你原有语言里没有的概念。

## 遗留问题

- 原作者为什么这么执着于 frozen？Python 里真的能保留那份"不可变"的味道吗？
- `parity_audit.py` 本身是个反讽——作者知道自己没写完。
